# Эмоции Гид: полное описание функционала проекта

Дата исследования: 23 мая 2026 года.
Актуализация: 6 июня 2026 года - добавлены исправления refresh-token flow, CORS, путей статических ассетов, единого хранения результатов в `Users/{uid}/TestResults` и передачи всех результатов тестов в AI-контекст.

Документ составлен по результатам изучения исходного кода frontend/backend и ручной проверки запущенного сайта через MCP Playwright по адресу `http://192.168.1.101:3001/`. Для live-проверки использовался специально созданный тестовый аккаунт. Пароли, токены и другие секреты в документ не включены.

## 1. Назначение проекта

`Эмоции Гид` - веб-приложение для самонаблюдения за эмоциональным состоянием. Основные сценарии:

- регистрация и вход пользователя;
- прохождение психологических тестов;
- сохранение результатов тестов в пользовательский профиль;
- просмотр динамики состояния на графиках;
- общение с ИИ-психологом и анализ эмоционального состояния по результатам тестов;
- использование дыхательных, медитативных и релаксационных упражнений;
- просмотр профиля и управление учетной записью;
- обратная связь и информационный блок "О нас".

Проект состоит из React/Vite frontend и FastAPI backend. Для авторизации и хранения данных используется Firebase.

## 3. Доступность сайта и конфигурация API

Проверенный сайт был доступен по адресу:

```text
http://192.168.1.101:3001/
```

В live-сборке frontend использовал backend API:

```text
http://localhost:8000
```

В `Frontend/src/services/api.ts` базовый URL берется из переменной окружения:

```text
VITE_API_BASE_URL
```

Если переменная не задана, используется fallback:

```text
https://emotions-guide.ru/
```

Frontend хранит состояние авторизации в `localStorage`:

- `authToken` - access token;
- `refreshToken` - refresh token;
- `userId` - идентификатор пользователя;
- `username` - имя пользователя.

На стороне frontend защищенные маршруты проверяют только наличие `authToken` в `localStorage`. При ответе API `401 Unauthorized` Axios interceptor очищает локальные токены и перенаправляет пользователя на `/auth`.

## 6. Авторизация и учетная запись

Маршрут: `/auth`.

Страница авторизации поддерживает два режима:

- вход в существующий аккаунт;
- регистрация нового аккаунта.

### Вход

Пользователь вводит:

- email;
- пароль.

После успешного входа frontend сохраняет токены и данные пользователя в `localStorage`, затем открывает авторизованную часть приложения.

Backend endpoint:

```text
POST /api/auth/login
```

Backend использует Firebase Identity Toolkit REST API `signInWithPassword`, получает Firebase ID token, создает refresh token и возвращает frontend:

- `accessToken`;
- `refreshToken`;
- `userId`;
- `email`;
- `username`;
- `expiresIn`.

### Регистрация

Пользователь вводит данные нового аккаунта. Backend создает пользователя в Firebase Auth и сохраняет профиль в Realtime Database.

Backend endpoint:

```text
POST /api/auth/register
```

В базе создается запись:

```text
Users/{uid}
```

Сохраняемые поля:

- `email`;
- `username`;
- `termsAccepted`;
- `createdAt`.

### Refresh token

Backend endpoint:

```text
POST /api/auth/refresh
```

Refresh-токен проверяется как project JWT и сверяется с blacklist `InvalidRefreshTokens/{hash}`. Если токен действителен, backend получает `uid`, создает Firebase custom token только как промежуточный токен, обменивает его через Firebase Identity Toolkit `signInWithCustomToken` на Firebase ID token и возвращает этот ID token как новый `accessToken`.

Одновременно генерируется новый project refresh token. Защищенные endpoint продолжают проверять access token через Firebase `verify_id_token`, поэтому refresh-сценарий совместим с обычной авторизацией.

### Logout

Backend endpoint:

```text
POST /api/auth/logout
```

При выходе refresh token хэшируется и записывается в Firebase Realtime Database:

```text
InvalidRefreshTokens/{hash}
```

После этого frontend должен очистить локальные токены.

### Смена пароля

Backend endpoint:

```text
POST /api/auth/change-password
```

Сценарий:

- пользователь передает текущий пароль и новый пароль;
- backend проверяет текущий пароль через Firebase Identity Toolkit;
- если текущий пароль верный, Firebase Auth обновляет пароль пользователя.

### Смена email

Backend endpoint:

```text
PUT /api/users/change-email
```

Endpoint присутствует в backend, но отдельного полноценного UI-сценария смены email в проверенном frontend не обнаружено. На странице профиля email отображается как read-only поле.

## 8. Психологические тесты

Маршрут: `/tests`.

Страница содержит список доступных тестов. Каждый тест отображается карточкой с названием, описанием и количеством вопросов. После выбора теста пользователь проходит вопросы последовательно.

Общее поведение тестов:

- вопросы показываются по одному;
- отображается текущий номер вопроса и общее количество вопросов;
- пользователь выбирает один вариант ответа;
- после выбора приложение переходит к следующему вопросу;
- после последнего ответа результат отправляется на backend;
- backend рассчитывает показатели, интерпретацию и рекомендации;
- результат сохраняется в базу;
- пользователь видит итоговую страницу результата.

### 8.1. Тест САН

Backend endpoints:

```text
GET /api/san/questions
POST /api/san/process
```

Тест содержит 30 вопросов. Каждый вопрос оценивается по шкале от 1 до 7:

- 1 - согласие с негативным полюсом;
- 4 - нейтральное состояние;
- 7 - согласие с позитивным полюсом.

Вопросы распределены по трем шкалам:

- `Самочувствие`;
- `Активность`;
- `Настроение`.

Расчет:

- по каждой шкале берется среднее значение;
- результат округляется до одного знака после запятой;
- итоговая интерпретация строится по среднему значению трех шкал.

Группы вопросов:

- самочувствие: индексы `0, 1, 6, 7, 12, 13, 18, 19, 24, 25`;
- активность: индексы `2, 3, 8, 9, 14, 15, 20, 21, 26, 27`;
- настроение: индексы `4, 5, 10, 11, 16, 17, 22, 23, 28, 29`.

Пороговая интерпретация:

- `>= 5.5` - отличное состояние;
- `>= 4.5` - хорошее состояние;
- `>= 3.5` - нормальное состояние;
- `>= 2.5` - сниженное состояние;
- `< 2.5` - состояние требует восстановления.

Дополнительное правило: если все три шкалы находятся на критически низком уровне, в рекомендации добавляется совет обратиться за профессиональной помощью и телефон доверия.

Сохранение результатов:

```text
Users/{uid}/TestResults
```

Поля результата:

- `wellbeingScore`;
- `activityScore`;
- `moodScore`;
- `interpretation`;
- `recommendations`;
- `timestamp`.

Live-проверка:

- при ответах `4` по всем 30 вопросам итоговые шкалы равны `4`, интерпретация - средний/нормальный уровень;
- при ответах `5` по всем 30 вопросам итоговые шкалы равны `5`, интерпретация - высокий/хороший уровень;
- оба результата появились в графиках.

Особенность live-поведения: после выбора ответа на последнем вопросе САН UI иногда оставался на последнем вопросе до дополнительного клика. Это похоже на проблему со stale state при завершении теста.

Compatibility score для общей формы результата:

```text
10 - total_score / 154 * 10
```

- `balance = positive_score - negative_score`;
- интерпретацию;
- рекомендации.

Диапазоны:

- positive score: от 5 до 25;
- negative score: от 5 до 25;
- balance: от -20 до 20.

Интерпретация balance:

- `> 10` - очень позитивное настроение;
- `> 0` - умеренно позитивное настроение;
- `> -10` - умеренно негативное настроение;
- `<= -10` - сильно негативное настроение.

Сохранение:

```text
Users/{uid}/TestResults
```

Замечание: в тексте интерпретации positive/negative score упоминается шкала `из 20`, хотя фактический диапазон при 5 вопросах и шкале 1-5 равен 5-25.

## 9. Хранение результатов тестов

В проекте используется единая структура хранения результатов.

### 9.1. Единая ветка результатов

Все тесты сохраняются в ветку:


Каждая запись содержит `testType`, `testName`, compatibility-поля `wellbeingScore`, `activityScore`, `moodScore`, объект `scores`, `interpretation` и `timestamp`.

Эту ветку используют:

Backend endpoint:

6. Сохраняет ответ ИИ.

3. Загружает последние результаты всех тестов из `Users/{uid}/TestResults`.
4. Передает данные в AIService.
5. Сохраняет ответ ИИ в историю чата.
6. Обновляет metadata анализа.

Для AI-анализа используется до 25 последних результатов, максимум 5 записей на один `testType`.

`POST /api/chat/messages` также добавляет в system prompt краткий контекст последних результатов всех тестов, но не сохраняет этот JSON-контекст в `ChatMessages`.

### 11.7. Актуальное состояние AI-функций

Ранее во время live-проверки две ключевые AI-операции возвращали ошибку backend. После исправления ручки ИИ-психолога работают корректно при наличии валидной конфигурации внешнего AI API:

## 16. Защита и ограничения доступа

### Frontend

Frontend защищает приватные маршруты через `ProtectedRoute`, который проверяет наличие `authToken` в `localStorage`.

При ошибке `401` Axios interceptor:

- при первом `401` и наличии `refreshToken` вызывает `POST /api/auth/refresh`;
- сохраняет новый `authToken`, новый `refreshToken` и `userId`;
- повторяет исходный запрос один раз с новым `Authorization: Bearer ...`;
- если refresh не сработал, удаляет `authToken`, `refreshToken`, `userId`, `username` и перенаправляет пользователя на `/auth`.

### Backend

Backend защищает приватные endpoint через проверку Firebase token.

CORS настроен через env-переменную `CORS_ALLOWED_ORIGINS` с безопасными local/prod defaults:

```text
http://localhost:3000
http://127.0.0.1:3000
http://localhost:5173
http://127.0.0.1:5173
https://emotions-guide.ru
https://www.emotions-guide.ru
```

SlowAPI подключен, но в найденных route-файлах явные per-route decorators rate limit не обнаружены. Основные пользовательские ограничения для AI реализованы вручную через cooldown.

### AI cooldown

Для AI-операций реализованы ограничения по времени:

- анализ эмоций нельзя запускать слишком часто;
- сообщения в чат нельзя отправлять слишком часто.

Metadata возвращает оставшееся время cooldown для чата и анализа. При `0` пользователь может инициировать соответствующее AI-действие.

## 19. Замеченные проблемы и ограничения

### 19.1. Последний ответ в САН может требовать дополнительного клика

Во время прохождения САН выбор ответа на последнем вопросе не всегда сразу завершал тест. Дополнительный клик по последнему варианту отправлял результат. Вероятная причина - использование устаревшего состояния ответов при проверке завершения теста.

### 19.2. Тренд в графиках может быть инвертирован

Live-проверка показала `Снижение`, хотя новый результат был выше предыдущего. Возможная причина - backend возвращает результаты от новых к старым, а расчет тренда ожидает хронологический порядок от старых к новым.

### 19.3. Период "Все" в графиках не является настоящей полной выборкой

В UI есть режим `Все`, но API-запросы используют ограниченный `limit`. Если у пользователя накопится много результатов, режим может не показать всю историю.

### 19.4. PSM-25 имеет недостижимый высокий уровень

При 25 вопросах и шкале 1-5 максимум равен 125. Код считает высокий стресс только при `> 154`, поэтому высокий уровень недостижим.

### 19.5. Шкала настроения содержит неточность в тексте интерпретации

Фактический диапазон positive/negative score равен 5-25, но текст интерпретации говорит `из 20`.

### 19.6. В тесте Бойко есть placeholder-вопросы

В исходниках обнаружены вопросы с placeholder-формулировками. Перед использованием в реальном психологическом сценарии нужно заменить их на валидированные вопросы.

### 19.7. Frontend-защита маршрутов основана только на localStorage

`ProtectedRoute` проверяет наличие токена, но не валидирует его до входа на страницу. Реальная защита обеспечивается backend, однако пользователь может попасть на защищенный route с устаревшим/невалидным токеном до первого API-запроса.

## 21. Что проект уже умеет полноценно

Проект уже предоставляет законченную пользовательскую оболочку для эмоционального дневника:
